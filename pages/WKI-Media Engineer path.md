			- ### **Domain 1: The Compression Layer (The "Physics")**
			  collapsed:: true
				- _This defines the math and logic of how we shrink images to fit through a wire._
					- **Key Codecs:** H.264 (AVC), H.265 (HEVC), VP9, AV1.
					- **The "Senior" Concepts:**
						- **GOP Structure:** Understanding how I-Frames (full images) relate to P/B-Frames (the "changes" between images).
						- **NAL Units (Network Abstraction Layer):** The actual "packets" inside a codec.
						- **Bitrate Control:** The difference between CBR (Constant), VBR (Variable), and **CRF** (Constant Rate Factor).
						- **Hardware vs. Software:** Knowing when to use a CPU (Libx264) vs. a GPU (NVENC/QuickSync).
			- ### **Domain 2: The Packaging Layer (The "Box")**
			  collapsed:: true
				- _This is how we wrap that compressed data so a player knows how to read it._
					- **Containers:** MP4, MPEG-TS, fMP4 (Fragmented MP4), MKV.
					- **Streaming Formats:** HLS (Apple), DASH (Industry Standard), CMAF (The "Unified" format), CMAF chunked streaming, LL HLS
					- **The "Senior" Concepts:**
						- **Manifests:** Understanding `.m3u8` and `.mpd` logic—how to "signal" different bitrates.
						- **Keyframe Alignment:** Why every bitrate version of a video must have the I-Frame at the _exact_ same millisecond.
						- **SCTE-35:** The "Markers" inside the box that tell a server to insert an ad.
			- ### **Domain 3: The Transport Layer (The "Movement")**
			  collapsed:: true
				- _This is how the data physically moves from a server to a screen._
					- **HTTP-Based (Latency: 2s–30s):** Standard HLS, DASH, Low-Latency HLS.
					- **UDP-Based (Latency: <1s):** **WebRTC** (Interactive), **SRT** (Secure Reliable Transport), RTMP (Ingest).
					- **The "Senior" Concepts:**
						- **Jitter Buffers:** The "waiting room" that smooths out shaky internet.
						- **ICE/STUN/TURN:** How two computers find each other through firewalls (crucial for WebRTC).
						- **Packet Loss Recovery:** Using **FEC** (Forward Error Correction) or **ARQ** (Retransmission) to fix broken video data.
			- ### **Domain 4: The Framework Layer (The "Engine")**
			  collapsed:: true
				- _The tools and APIs we use to build the logic._
					- **C++ Level:** **FFmpeg (libavcodec/libavformat)**, GStreamer.
					- **Browser Level:** **WebCodecs API**, WebAssembly (WASM).
					- **The "Senior" Concepts:**
						- **Demuxing:** Taking a file apart to get the raw video/audio streams.
						- **Memory Management:** How to move a video frame from a C++ buffer to a GPU texture without crashing the computer.
						- **Thread Synchronization:** Making sure the audio doesn't drift away from the video.
			- ### **Domain 5: The Infrastructure Layer (The "Scale")**
			  collapsed:: true
				- _How we serve millions of people at the same time._
					- **The Stack:** AWS/GCP, Kubernetes (K8s), CDNs (Akamai, CloudFront).
					- **The "Senior" Concepts:**
						- **Edge Compute:** Running logic (like DRM checks or ad-insertion) at the CDN level instead of the main server.
						- **Distributed Transcoding:** Splitting one 2-hour movie into 100 small pieces to encode them all in 5 minutes.
						- **Observability:** Using **VMAF** (Netflix's metric) to automatically check if a video "looks good" without a human watching it.
			- ### **Domain 6: Real-Time Architecture (The "Interactive")**
			  collapsed:: true
				- _The highest level of system design for "live" logic._ This domain is actually a **synthesis of Domain 3 (Transport) and Domain 5 (Infrastructure).**
					- **SFU (Selective Forwarding Unit):** The server just "routes" video packets (Zoom/Google Meet style). Extremely fast.
					- **MCU (Multipoint Control Unit):** The server "mixes" all videos into one single video. Heavy on CPU, but works on weak devices.
			- ### Domain 7: Observability & Telemetry, QoE Analytics(The "Vitals")
			  collapsed:: true
				- _This is how we know the patient (the streaming system) is alive and healthy._
				- **The Data:** Not video frames, but **metadata about video delivery**—playback start times, buffering events, bitrate switches, error codes, CDN cache hits/misses, license server latencies.
				- **The Flow:**
					- **Producers:** Players (JavaScript, iOS, Android), encoders, packagers, CDN edges
					- **Transport:** Often a **data streaming platform** like Kafka, Kinesis, or Pub/Sub (yes, this is where it enters!)
					- **Processors:** Flink, Spark, or Lambda functions that aggregate, filter, and enrich
					- **Storage:** Time-series databases (InfluxDB, Prometheus), data warehouses (BigQuery, Snowflake)
					- **Consumption:** Dashboards (Grafana), alerting (PagerDuty), business intelligence
				- **The Senior Concepts:**
					- **Synthetic Monitoring:** Running "fake viewers" that constantly watch your stream to detect issues before real users do.
					- **QoE Scorecards:** Turning raw metrics (buffering %, join time) into a single "health score" per user session.
					- **Root Cause Correlation:** Automatically linking "high buffering in Brazil" to "CDN edge in Sao Paulo had cache miss spike" to "origin was overloaded at same time."

- ### **Codec & Container "Deep Dive" (The "Science" Layer)**
  collapsed:: true
	- _Target Jobs: All_ You know _how_ to use them; now you need to understand _why_ they work.
		- **Inter-frame vs. Intra-frame Compression:**
			- **I-Frames:** Full images (big size).
			- **P/B-Frames:** Only the "changes" between images (small size).
		- **Codec Evolution:** Know the trade-offs.
			- **H.264 (AVC):** Universal, but old.
			- **H.265 (HEVC):** 50% better compression, great for 4K, but has licensing fees.
			- **AV1:** The future. Royalty-free and even better than H.265, but very "heavy" to encode.
		- **Bitrate Control:** Understand **CBR** (Constant), **VBR** (Variable), and **CRF** (Constant Rate Factor). This is how you "Improve video quality while minimizing bandwidth".
		-
- ###  Real-Time Media Transport (The "Live" Layer)
  collapsed:: true
	- _Target Jobs: Stream, Euroka, Challenz, Squad_ This is the "Low Latency" world. You move from HTTP-based streaming (seconds of delay) to UDP-based streaming (milliseconds of delay).
		- **WebRTC Internals:** Don't just use a library. Understand **ICE, STUN, and TURN** (how two computers find each other through firewalls).
		- **Signaling:** Understand how to exchange "handshakes" using WebSockets or HTTP to start a call.
		- **SRT (Secure Reliable Transport):** This is the modern replacement for RTMP. Learn how it handles "packet loss" on shaky internet connections using **ARQ** (Automatic Repeat Request).
		- **SFU vs. MCU Architecture:** Learn why an **SFU** (Selective Forwarding Unit) is better for scaling 100 people in a video call compared to an MCU (which mixes everything on the server).
- ### **Advanced Metadata & Monetization (The "Professional" Layer)**
  collapsed:: true
	- _Target Jobs: Stand8, OTTera-style roles_ You already have a head start here, but need to "harden" it.
		- **DRM Deep Dive:** Move beyond "using" Widevine. Learn about **Microsoft PlayReady** and how license servers actually issue keys.
		- **SCTE-35 & SSAI:** Understand the specific markers inside a stream that tell a server "Put an ad here now."
		- **CMAF (Common Media Application Format):** This is the industry "holy grail"—one file format that works for both Apple (HLS) and Android/Web (DASH).
		-
- ### **Browser-Based Media Engines (The "Slate" Layer)**
  collapsed:: true
	- _Target Jobs: Slate_ This is about bringing the power of C++ and FFmpeg into the web browser.
		- **WebAssembly (WASM) & Emscripten:** Learn how to "compile" a C++ library into a format a browser can run. You don't need to be a C++ pro, but you need to understand how to "glue" JavaScript to a WASM module.
		- **WebCodecs API:** This is a new browser API that gives you direct access to the computer's hardware encoder/decoder. It’s much faster than using a `<video>` tag for things like video editors.
		- **OffscreenCanvas & Workers:** Learn how to process video frames in the "background" of a browser so the website doesn't freeze.
